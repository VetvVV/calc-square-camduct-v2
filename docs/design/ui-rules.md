# UI Rules

## General
- Build a professional engineering tool, not a SaaS landing dashboard.
- Keep the workspace wide and practical.
- Avoid decorative noise and oversized empty hero spaces in operational screens.
- Use plain, precise UI labels.
- Keep CAMduct/internal keys hidden from regular users unless a service/admin mode explicitly reveals them.

## Layout
- Home can be informative and brand-led.
- Atlas should behave like a technical visual catalog.
- Workspace should prioritize specification width and calculator clarity.
- Mobile may stack panels, but desktop should use available width.

## Color
- Use neutral light backgrounds.
- Use orange for primary actions, active state, and section accents.
- Use graphite for text and table headers.
- Do not create one-note palettes or heavy gradients.

## Tables
- Tables must be readable at production density.
- Keep header contrast clear.
- Align numeric values consistently.
- Avoid unnecessary card nesting around tables.

## Forms
- Labels must be short and clear.
- Inputs must have obvious focus state.
- Use select controls for finite options.
- Use disabled states only when paired with explanation for locked role features.

## Accessibility
- All interactive elements need keyboard focus visibility.
- Color cannot be the only state indicator.
- Role-locked functions should explain why they are unavailable.
